# EvalOps Lite 指标与评测体系

## 1. 指标框架

EvalOps Lite 将 AI 应用评测拆成四类：

- 质量：回答是否正确、完整、相关。
- 风险：是否幻觉、越权、安全违规。
- 体验：是否帮助用户完成任务。
- 成本：延迟和 token 成本是否可接受。

## 2. 北极星指标

每周通过门禁并完成灰度发布的 AI 功能版本数。

原因：

- 体现产品对真实迭代效率的贡献。
- 同时要求质量达标和上线落地。

## 3. 质量指标

| 指标 | 定义 | 示例目标 |
| --- | --- | --- |
| 准确率 | 与标准答案一致的比例 | >= 85% |
| 相关性 | 是否回答用户问题 | >= 90% |
| 完整性 | 是否覆盖标准答案要点 | >= 80% |
| 格式遵守率 | 是否符合要求输出格式 | >= 95% |
| 引用命中率 | 引用是否支持结论 | >= 85% |

## 4. 风险指标

| 指标 | 定义 | 示例目标 |
| --- | --- | --- |
| 幻觉率 | 无依据或错误事实比例 | <= 3% |
| 安全违规率 | 输出违规内容比例 | 0 |
| 拒答正确率 | 应拒答时正确拒答比例 | >= 90% |
| 高风险通过率 | 高风险样本评测通过比例 | >= 95% |
| 人工抽检不通过率 | 人审发现问题比例 | <= 5% |

## 5. 成本和工程指标

| 指标 | 定义 | 示例目标 |
| --- | --- | --- |
| P95 延迟 | 95% 请求响应时间 | <= 8 秒 |
| 单次成本 | 单次请求平均 token 成本 | 低于预算 |
| 失败率 | API 或流程失败比例 | <= 1% |

## 6. 评测方式

### 规则评测

- JSON 格式是否正确。
- 是否包含引用。
- 是否触发拒答。
- 是否命中关键词。

### LLM-as-Judge

适用于：

- 相关性。
- 完整性。
- 引用是否支撑结论。
- 是否存在幻觉。

### 人工抽检

适用于：

- 高风险样本。
- Judge 不确定样本。
- 新版本门禁前抽检。

## 7. 上线门禁模板

一个 AI 功能版本上线前必须满足：

- 核心测试集准确率 >= 85%。
- 高风险样本通过率 >= 95%。
- 幻觉率 <= 3%。
- 引用命中率 >= 85%。
- P95 延迟 <= 8 秒。
- 单次成本低于预算。
- 高风险样本人工抽检完成率 = 100%。

## 8. 失败归因标签

- `retrieval_miss`：检索没有召回正确上下文。
- `citation_wrong`：引用不能支撑结论。
- `hallucination`：模型编造内容。
- `instruction_following`：没有遵守格式或指令。
- `refusal_error`：该拒答未拒答，或不该拒答却拒答。
- `safety_violation`：安全或合规违规。
- `latency_cost`：延迟或成本超过门槛。

## 9. 面试表达

> EvalOps Lite 的核心不是做一个漂亮 dashboard，而是把 AI 功能上线标准产品化。PM 要定义什么叫“可用”、什么叫“风险不可接受”，并让每次 Prompt、模型或 RAG 参数变化都有可比较的评测结果。
